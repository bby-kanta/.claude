# Context Placement Strategy

プロジェクト知識を「必要なときに、必要な分だけ」Claudeのコンテキストに載せるための配置戦略ガイド。

CLAUDE.mdに何でも書くとコンテキストウィンドウを圧迫し、他のタスクの精度に悪影響を与える。配置先を間違えると、知識が無駄にトークンを消費するか、必要なときに見つからない。このスキルはその判断を体系化する。

**書く前のリトマステスト**: 「この1行を消したら、Claudeは間違えるか？」— Noなら書かない。コードやフレームワークのドキュメントから自明なこと、Claudeが推論で到達できることは、コンテキストに載せても精度向上に寄与しない。書くべきは「知らなければ間違える暗黙知」だけ。

## 4つの配置先とコンテキストコスト

```
コンテキストコスト:  高 ←――――――――――――――――――――→ 低
                 CLAUDE.md  Rules      SKILL   コード内
                 (root)     (paths付)          コメント
ロードタイミング:  常時     ファイル    タスク   ファイル
                            Read時   トリガー時  Read時
```

コストが高い配置先ほど「常にコンテキストに載る」ため発見されやすいが、その分トークンを常時消費する。コストの低い配置先は必要なときだけロードされるため効率的だが、トリガー条件を満たさないと見落とされる。

この順番を意識して、「本当に常時必要か？」を問いかけながら配置先を決める。

## 各配置先の詳細

### 1. CLAUDE.md（ルート） — 常時ロード・最高コスト

以下の **すべて** を満たす場合のみCLAUDE.mdに書く:

- プロジェクト全体に横断的に適用されるルール
- 頻繁に（月に複数回以上）関わる領域
- 1-2行で表現可能な簡潔なルール
- 特定のタスクに限定されない（「Xするとき」ではなく「常にXする」形式）

**良い例**: `Date.today ではなく Time.zone.today を使うこと（タイムゾーン不整合防止）`

**悪い例**: 特定のモデルの特定メソッド変更時の注意事項（→ paths付きRulesが適切）

CLAUDE.mdが肥大化してきた場合は、内容をpaths付きRulesやSKILLに移動して200行以内に保つ。

### 2. Rules — paths付き（条件付き） — ファイルRead時ロード

該当ファイルをClaudeがReadしたときだけコンテキストに載る。コスト効率が最も高い配置先の一つ。

```yaml
---
paths:
  - "app/models/order.rb"
  - "spec/models/order_spec.rb"
---
```

使うべき場面:
- 特定のファイルやファイル群を編集するときに意識すべきルール
- 複数ファイルにまたがるが、globパターンで対象を特定できる
- タスク種別ではなく「どのファイルを触るか」で発動すべき知識
- コード内コメントより広い文脈（背景や関連ファイルの一覧）が必要

**重要な制限**: paths付きRulesはClaudeがファイルをReadしたときにロードされる。Write（新規作成）時にはロードされない。レビュー用途では対象ファイルは必ずReadされるため問題ない。

### 3. SKILL — タスクトリガー時ロード

`.claude/skills/*/SKILL.md` に配置。スキルがトリガーされたときだけロードされる。

使うべき場面:
- 特定のタスク種別にのみ必要な知識（「API endpoint作成」「デプロイ手順」など）
- 複数ステップのチェックリストや手順書
- 詳細な説明が必要（ファイルパス、具体的な配列名、grepコマンドなど）
- 頻度が低い（月1回未満）がミスすると影響が大きい

**RulesとSKILLの違い**:
- **Rules**: 「このファイルを触るときはこれを意識して」→ ファイルベースのトリガー
- **SKILL**: 「このタスクをやるときはこの手順で」→ タスクベースのトリガー

### 4. コード内コメント — コンテキストコストゼロ

そのファイルが読まれたときに自然に目に入る。Claude・AI・人間すべてのレビュアーに等しく効く。

使うべき場面:
- 特定のメソッドや定数に紐づく注意事項
- 「このXを変更したらYも変更すること」という1対1の依存関係
- コードの近くにあることが重要（離れた場所のルールでは見逃されやすい）

**例**: `# NOTE: この配列に要素を追加したら spec/models/order_spec.rb の expected_reasons も更新すること`

## 判断フローチャート

知識をどこに配置するか迷ったら、上から順に判断する:

```
このルールは特定のタスク種別にのみ必要か？（例: デプロイ手順、マイグレーション作成）
  ├─ Yes → SKILL に書く
  └─ No
      ↓
このルールは特定のファイル群に紐づくか？
  ├─ Yes → globパターンで対象ファイルを特定できるか？
  │         ├─ Yes → paths付き Rules に書く ← 多くのケースでここが最適
  │         │         + 補助的にコード内コメントも追加（防御的多層化）
  │         └─ No → コード内コメントを推奨
  └─ No
      ↓
プロジェクト全体に常時適用すべきか？
  ├─ Yes → CLAUDE.md（1-2行で簡潔に。肥大化したらpaths付きRulesやSKILLに移動）
  └─ No → SKILL として切り出す
```

---

## 防御的多層化（推奨パターン）

1つの知識に対して複数の配置先を組み合わせてよい。これは冗長ではなく防御的多層化。ただし同じ内容をCLAUDE.mdとpaths付きRulesの両方に書くのはコンテキストの二重消費なので避ける。

**組み合わせの例**:
- **paths付きRules** + **コード内コメント**: Rulesでファイル間の依存関係を説明し、コメントでピンポイントの注意喚起。最も効果的な組み合わせ
- **SKILL** + **コード内コメント**: SKILLで手順全体を管理し、コメントで個別の注意点をマーク
- **CLAUDE.md** + **paths付きRules**: CLAUDE.mdに1行の原則を書き、Rulesに詳細な適用ルールを記載

## よくある間違い

| やりがち | なぜ問題か | 正しい配置 |
|---------|-----------|-----------|
| 全部CLAUDE.mdに書く | コンテキスト肥大化で全体の精度低下 | フローチャートで判断 |
| 低頻度ルールをCLAUDE.mdに書く | 月1回の作業のために常時トークン消費 | SKILL or paths付きRules |
| ファイル固有の注意をSKILLに書く | ファイルを触ってもSKILLはトリガーされない | paths付きRules + コード内コメント |
| コード内コメントだけに頼る | 新規ファイル作成時はコメントが存在しない | paths付きRules + コード内コメントの併用 |
| CLAUDE.mdを200行超にする | 後半のルールが効きにくくなる | paths付きRulesやSKILLに移動 |
