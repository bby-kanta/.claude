# PR / コミット レビュースキル

## 引数のパース

`$ARGUMENTS` を以下のルールでパースする：

| 入力パターン | モード | 例 |
|---|---|---|
| `#数字` または `数字` のみ | PR番号 | `#7783`, `7783` |
| `by_hash <hash>` | コミットハッシュ | `by_hash abc1234` |
| 引数なし | 現ブランチのPR | （自動検出） |

引数にレビュー観点の補足（例: `#7783 パフォーマンス重点で`）が含まれる場合は、数値部分だけをPR番号として取り出し、残りはレビュー時の重点観点として活用する。

---

## Step 1: 情報取得

### GitHub CLI の実行について

**`gh` コマンドを直接実行してはいけない。** macOSサンドボックスがキーチェーンへのアクセスをブロックし、TLS証明書検証エラー（`x509: OSStatus -26276`）で失敗する。

代わりに `/gh` スキルを使用する。`/gh` スキルは `dangerouslyDisableSandbox: true` を設定して `gh` コマンドを実行するため、サンドボックス制約を回避できる。

### PR番号モード

`/gh` スキル経由で以下を実行：
```
gh pr view <PR番号> --json title,body,baseRefName,headRefName,author,labels,files,commits
gh pr diff <PR番号>
```

### by_hash モード

コミットハッシュの場合は `gh` 不要。通常の Bash で実行：
```bash
git show <hash> --stat
git diff <hash>^..<hash>
```

### 引数なしモード

`/gh` スキル経由で以下を実行：
```
gh pr view --json number,title,body,baseRefName,headRefName,author,labels,files,commits
```
で現ブランチのPR情報を取得し、PR番号を使って `gh pr diff` を実行する。

## Step 2: 変更の文脈を理解する

diffだけでは正確なレビューができない。以下を行う：

1. **変更ファイルの元コードを読む** — `Read` ツールで変更前のファイル全体を読み、変更が既存のコードベースにどう馴染むかを把握する
2. **関連ファイルを確認する** — 変更されたクラスの親クラス、呼び出し元、関連するテストファイルなど
3. **PR description を読む** — 変更の意図・背景を理解し、意図に沿ったレビューをする

特に以下のファイルは重点的に読む：
- モデル・サービス・コントローラーの本体
- 対応するspecファイル
- 変更に関連するルーティング・スキーマ定義

---

## Step 3: レビュー

以下の観点でレビューする。**問題がある箇所だけ** を指摘し、問題がなければ余計なコメントは不要。

| 観点 | 具体的に見ること |
|---|---|
| **バグ・ロジック** | off-by-one、nil安全性、race condition、境界値、例外ハンドリング漏れ |
| **セキュリティ** | SQLインジェクション、mass assignment、認可漏れ、秘密情報のハードコード |
| **パフォーマンス** | N+1クエリ、不要なeager load、インデックス不足、大量データでのスケーラビリティ |
| **設計** | 責務の分離、既存パターンとの一貫性、不要な複雑さ |
| **テスト** | カバレッジの十分さ、境界値・異常系テスト、テストの独立性 |
| **命名・可読性** | 変数名・メソッド名が意図を伝えているか、マジックナンバー |

レビューで意識すること：
- 「自分ならこう書く」という好みの押し付けではなく、**実害がある問題** にフォーカスする
- 既存コードのスタイルに合っているなら、そのスタイルを尊重する
- テストが十分かは変更の影響範囲で判断する（軽微な修正に網羅的テストを求めない）

## Step 4: 出力フォーマット

```markdown
## PR #<番号>: <タイトル>
（by_hashモードの場合: `## Commit <hash>: <コミットメッセージ>`）

### 概要
<変更内容を1-2文で要約>

### 指摘事項

#### `<ファイルパス>`:<行番号付近>
- **🔴 must** | **🟡 should** | **🔵 nit**: 指摘内容
  - 理由: なぜこれが問題か
  - 提案: こうすると良い（コード例があれば添える）

（指摘がなければこのセクションは省略）

### 総評
<全体の評価。問題なければ「LGTM」の一言で良い>
```

### 重要度の基準

| レベル | 意味 | 例 |
|---|---|---|
| 🔴 **must** | マージ前に修正すべき | バグ、セキュリティ脆弱性、データ不整合、本番障害リスク |
| 🟡 **should** | 強く推奨するが判断は任せる | パフォーマンス懸念、設計上の問題、テスト不足 |
| 🔵 **nit** | 余裕があれば | 命名改善、軽微なリファクタ、スタイル |
