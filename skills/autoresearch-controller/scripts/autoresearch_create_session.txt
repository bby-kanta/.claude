⏺ ---
  karpathy/autoresearch 詳細分析

  概要

  Andrej Karpathy（元Tesla AI Director、OpenAI共同創設者）が2026年3月に公開した
  リポジトリ。AIエージェントに小規模LLMの訓練を自律的に繰り返させ、人間が寝てい
  る間にモデルを改善させるというプロジェクト。公開わずか3週間で
  約60,000スター、8,300フォーク
  を獲得しており、コミュニティの関心の高さがうかがえます。

  ---
  あなたの所感に対する評価

  ▎ AIに何度も同じ課題を与えそれに取り組むことをloopし、これまでの試行からAIが自
  律的に判断し、改善のサイクルを回す

  これは概ね正確です。ただし補足すると：

  - 「同じ課題」というより、同一の評価基準（val_bpb、5分固定）のもとで、毎回異な
  るアプローチを試すのが正確
  - AIが過去の試行結果（results.tsvとgit履歴）を参照し、何が効いて何がダメだった
  かを自分で判断する
  - 改善したら keep（ブランチを進める）、悪化したら discard（git reset
  で巻き戻す）という進化的な選択圧がかかっている

  ---
  アーキテクチャ設計：3ファイル構成の意味

  prepare.py  ── 固定された「ルール」（データ、評価、定数）── 変更不可
  train.py    ── AIが自由に改変できる「実験対象」         ── AIが編集
  program.md  ── AIへの指示書（＝研究組織の設計図）       ── 人間が編集

  1. prepare.py（不変の基盤 / 審判役）

  - 定数: MAX_SEQ_LEN=2048, TIME_BUDGET=300（5分）, EVAL_TOKENS=40*524288
  - データ:
  HuggingFaceからClimbMix-400Bデータセットのparquetシャードをダウンロード
  - トークナイザ: rustbpeで8192語彙のBPEを学習、tiktokenフォーマットで保存
  - データローダ: BOS-alignedのbest-fit packing方式（パディングなし100%利用率）
  - 評価関数 evaluate_bpb(): bits per byte (BPB)
  を算出。語彙サイズ非依存なので、アーキテクチャ変更しても公平に比較可能

  設計意図: 実験の「ルール」を固定することで、AIが評価基準自体をハックするのを防
  ぐ。科学実験における制御変数。

  2. train.py（AIの実験場 / 約450行）

  AIが自由に改変できる唯一のファイル。中身は：

  モデル（GPT）:
  - Transformer decoderベースのGPTモデル
  - Value Embedding (ResFormer): 交互レイヤーにvalue
  residual接続。入力依存のゲート付き
  - RoPE (Rotary Position Embedding): 位置エンコーディング
  - Flash Attention 3: H100 (Hopper) では varunneal版、非Hopperでは
  kernels-community版を自動選択
  - Sliding Window Attention: SSSLパターン（3層Short + 1層Long の繰り返し）
  - Logit softcap: 15 * tanh(logits/15) でlogitを制限
  - Per-layer residual/x0 lambdas: 層ごとの学習可能なスカラー（residual
  stream制御）
  - 活性化関数: relu(x)^2（ReGLUではなくsquared ReLU）

  オプティマイザ（MuonAdamW）:
  - Muon: 2D行列パラメータ用。Polar Express直交化 + NorMuon分散削減 + Cautious
  Weight Decay
  - AdamW: embedding、unembedding、スカラーパラメータ用
  - パラメータ種別ごとに異なる学習率（embedding=0.6, unembedding=0.004,
  matrix=0.04, scalar=0.5）
  - torch.compile で高速化

  ハイパーパラメータ:
  - DEPTH=8, ASPECT_RATIO=64 → model_dim = 512
  - TOTAL_BATCH_SIZE=2^19 (524K tokens/step)
  - Warmup 0%, Warmdown 50%, Final LR 0%
  - 約50Mパラメータ

  訓練ループ:
  - 最初の10ステップはウォームアップ（コンパイル時間除外のため計時しない）
  - EMAでsmoothed training lossを表示
  - NaN/loss>100で即座にFAIL終了
  - GCを無効化（Python GCの500msストール回避）

  3. program.md（AIエージェントへの指示書）

  これが最も革新的な部分。人間が書く「メタプログラム」で、AIの研究行動を定義する
  ：

  セットアップフェーズ:
  1. 日付ベースのブランチ作成（autoresearch/mar5）
  2. リポジトリの全ファイルを読ませてコンテキスト把握
  3. データ存在確認
  4. results.tsv の初期化
  5. ベースライン実行

  実験ループ（永続）:
  LOOP FOREVER:
    1. git状態確認
    2. train.pyを実験的に変更
    3. git commit
    4. 実行: uv run train.py > run.log 2>&1
    5. 結果抽出: grep "^val_bpb:" run.log
    6. クラッシュ時はtail -n 50でスタックトレース確認→修正試行
    7. results.tsvに記録
    8. 改善 → keep（ブランチ進行）
    9. 悪化 → discard（git reset）

  重要なルール:
  - NEVER STOP: 人間が手動で止めるまで永遠にループ。「続けていい？」と聞くな
  - 簡潔性基準:
  改善が小さくてコードが複雑になるなら不採用。コード削除で同等性能なら採用
  - タイムアウト: 10分超えたら強制終了
  - VRAM: ソフト制約（意味のある改善なら多少の増加OK）

  2. 「program.md」= 研究組織のソースコード

  Karpathyの最も重要な洞察は：

  ▎ The core idea is that you're not touching any of the Python files like you 
  normally would as a researcher. Instead, you are programming the program.md 
  Markdown files that provide context to the AI agents and set up your 
  autonomous research org.

  つまり、人間の役割は：
  - Pythonコードを書くことではなく
  - AIエージェントの行動原理を設計すること（＝組織設計、マネジメント）

  program.mdを改善することは、研究チームのマネジメント手法を改善することに等しい
  。

  3. 進化的アルゴリズムの暗黙的実装

  mutation  = AIがtrain.pyを変更
  fitness   = val_bpb（低いほど良い）
  selection = keep or discard
  elitism   = 常に最良の個体（コード）からスタート

  これは実質的に (1+1)進化戦略 に似ている。ただし突然変異がランダムではなく、AI
  の「理解」に基づく知的な変更である点が決定的に異なる。

  4. 固定時間予算の設計思想

  5分固定にした理由が2つある：
  1. 公平性:
  モデルサイズ、バッチサイズ、アーキテクチャを変えても同一基準で比較可能
  2. プラットフォーム最適化: そのGPUで5分間で最も良い結果を出すモデルを見つける

  トレードオフとして、異なるGPU間で結果の比較ができなくなる（H100とRTX
  4090で全く異なる最適解になりうる）。

  ---
  技術的詳細

  BPB (Bits Per Byte) メトリクスの選択理由

  通常のperplexityや cross-entropy loss はトークナイザの語彙サイズに依存する。au
  toresearchではAIが語彙サイズ自体を変更する可能性があるため、バイトあたりのビッ
  ト数という語彙非依存のメトリクスを採用。計算方法：

  BPB = Σ(per-token cross-entropy in nats) / (ln(2) × Σ(target token bytes))

  Muonオプティマイザ

  最先端のオプティマイザで、行列パラメータに対して：
  1. Nesterov momentum
  2. Polar Express直交化: 勾配を直交行列に射影（NewtonSchulz反復の多項式近似）
  3. NorMuon: second momentによる分散削減
  4. Cautious weight decay: 勾配とパラメータの符号が一致する場合のみweight
  decayを適用

  torch.compile の活用

  - モデル全体をtorch.compile(dynamic=False)
  - AdamWステップとMuonステップも個別に@torch.compile(dynamic=False, 
  fullgraph=True)
  - 0次元CPUテンソルでハイパーパラメータを渡し、値変更時の再コンパイルを回避

  計測方法について

  Railsには Benchmark モジュール（Ruby標準ライブラリ）があります。特別なGem不要
  で使えます。ただし、コントローラーのレスポンスタイムを計測するなら、もう少し実
  践的なアプローチがあります：

  # Railsの統合テスト基盤を使ったベンチマーク
  require 'benchmark'

  N = 20
  result = Benchmark.measure do
    N.times { get "/users/1" }
  end
  puts "avg: #{(result.real / N * 1000).round(1)}ms"

  あるいは rack-mini-profiler なしでも ActiveSupport::Notifications
  でSQLクエリ数・時間が取れます。

  提案: autoresearchの prepare.py に相当するベンチマークランナースクリプトを用意
  し、これをAI編集不可にする。AIはスクリプトを実行して結果だけ見る。

  ---
  核心的な設計判断：テストデータの盲検化

  これはあなたが指摘した中で最も重要な論点です。autoresearchでは prepare.py
  が評価データを内部的に管理し、AIは evaluate_bpb()
  を呼ぶだけ。同じ構造が必要です。

  問題:
  AIがテストデータの中身を読めると、そのデータに特化した最適化をしてしまう（例：
  「ユーザーが100人しかいない」と知って、100人前提のハードコード最適化をする）

  現実的なアプローチ（案）:

  benchmark_runner.rb   ← AI編集不可・読み取り不可（データ生成 + 計測ロジック）
  benchmark_config.yml  ← AI編集不可（エンドポイント、N回数、閾値だけ定義）

  ただし、ハードに読み取りを防ぐのは不可能です。ファイルシステムにある以上、AIは
   Read ツールで読めます。autoresearchも同様で、prepare.py
  の読み取り禁止は「指示ベース」のソフト制約です。

  ここで2つの選択肢があります：

  A. ソフト制約（autoresearch方式）
  - スキルの指示で「benchmark_runner.rb
  の中身は読むな、実行して結果だけ見ろ」と明記
  - シンプル、実用的。ほとんどの場合これで十分

  B. アーキテクチャ的分離
  - ベンチマークランナーがテストデータをランタイムに動的生成する（seed付きランダ
  ム）
  - AI がファイルを読んでも、実行時のデータは分からない
  - より堅牢だが、テストデータの一貫性がやや弱まる（seedで固定は可能）

  どちらが好みですか？
  個人的にはAで始めて、必要ならBに進化させるのが良いと思います。

  ループの流れ

  LOOP:
    1. 現状のgit状態・過去のresultsを確認

    3. git commit
    4. db:migrate（インデックス追加時）
    5. 関連specを実行 → FAILならrevert
    6. ベンチマーク実行 → 結果取得
    7. results.tsvに記録
    8. 改善 → keep / 悪化 → git reset

  防御機構

  - ブランチ保護: 初回セットアップで perf-tune/xxx
  ブランチを切り、mainにいたら即abort
  - 変更範囲制限:
  指定されたcontroller/model/service以外のファイルを触ったらrevert
  - spec必須パス: 変更に関連するspecが通らなければ即discard
  - ベンチマーク改竄防止: benchmark_runner.rb のdiffがあればabort

  1. テストデータ盲検化：二重エージェント + Hooks

  あなたのアイデアを整理するとこうなります：

  ┌─────────────────────────────────────┐
  │  Data Setup Agent (初回 or 再seed)   │
  │  ・benchmark/seed_data.rb を読める   │
  │  ・DBにテストデータをINSERT          │
  │  ・スキーマ変更時に再実行            │
  └─────────────────────────────────────┘
            ↓ データ準備完了
  ┌─────────────────────────────────────┐
  │  Tuning Agent (ループ本体)           │
  │  ・controller/model/service を編集   │
  │  ・benchmark/seed_data.rb 読み取り禁止│  ← Hooks + ソフト制約
  │  ・benchmark/run.rb を実行のみ       │
  │  ・出力されたメトリクスだけ見る      │
  └─────────────────────────────────────┘

  Hooksの具体的な実装イメージ：

  // .claude/settings.json
  {
    "hooks": {
      "PreToolUse": [
        {
          "matcher": "Read",
          "command": "bash -c '[ \"$TOOL_INPUT\" != *benchmark/seed_data* ] && [
   \"$TOOL_INPUT\" != *benchmark/run.rb* ] || (echo \"BLOCK:
  benchmark内部ファイルの読み取りは禁止です\" && exit 1)'"
        }
      ]
    }
  }

  ただしHooks はセッション全体に適用されるので、Data Setup
  Agentも同じ制約を受けます。これの回避策は：

  - Data Setup Agentは Bashツール経由 でスクリプトを実行する（ruby 
  benchmark/setup.rb）
  - setup.rb自体がseed_data.rbをruby内でrequireするので、Hooksの Read
  ツール制約を迂回できる
  - Tuning Agentもベンチマーク実行は ruby benchmark/run.rb
  をBashで実行するので問題なし

  つまり Read ツールでの直接閲覧をHooksでブロック + Bash経由の実行は通す
  という設計です。

  ソフト制約（SKILL.mdに記載）：
  ▎ ベンチマークのテストデータの内容を推測・分析してはならない。cat、head、grep 
  等でbenchmarkディレクトリ内のデータファイルを読み取る行為も禁止。あなたが最適
  化すべきはコードのパフォーマンスであり、特定のデータに対する最適化ではない。

  4. /loop連携の設計

  /loopの設定時間次第というのは正しいです。スキルの責務を明確に分けましょう：

  ┌─────────────────────────────┬──────────────────────────────────┐
  │            責務             │               担当               │
  ├─────────────────────────────┼──────────────────────────────────┤
  │ 「1サイクルとは何か」の定義 │ SKILL.md                         │
  ├─────────────────────────────┼──────────────────────────────────┤
  │ 何回・何分間回すか          │ /loop コマンド                   │
  ├─────────────────────────────┼──────────────────────────────────┤
  │ いつ止めるか                │ ユーザー or /loop のタイムアウト │
  └─────────────────────────────┴──────────────────────────────────┘

  1サイクルの所要時間の見積もり：
  - AIの思考・コード変更: ~30秒〜1分
  - migration実行: ~5秒（あれば）
  - 関連spec実行: ~30秒〜2分（プロジェクト次第）
  - ベンチマーク実行: ~30秒〜1分（N回リクエスト）
  - 合計: 約2〜5分/サイクル

  なので /loop 6m /controller-tuner くらいが妥当。

  重要な懸念点: /loop間隔 < サイクル時間 だと、前のサイクルが終わる前に次が起動
  して同じファイルを2エージェントが同時編集する危険があります。対策として：

  # ロックファイル方式（SKILL内でサイクル開始時に取得）
  if [ -f .tuning.lock ]; then
    echo "前のサイクルが実行中です。スキップします。"
    exit 0
  fi
  touch .tuning.lock
  # ... サイクル実行 ...
  rm .tuning.lock

  サブエージェント別Hooks

  .claude/agents/*.md
  のフロントマターにhooksを定義でき、そのエージェントの実行中だけ適用されます。

  具体的な設計

  .claude/agents/tuning-agent.md:
  ---
  name: tuning-agent
  description: コントローラーのパフォーマンスチューニングを実行する
  tools: Read, Edit, Bash, Glob, Grep
  hooks:
    PreToolUse:
      - matcher: "Read"
        hooks:
          - type: command
            command: "$CLAUDE_PROJECT_DIR/.claude/hooks/block-benchmark-read.sh"
  ---

  .claude/agents/data-setup-agent.md:
  ---
  name: data-setup-agent
  description: ベンチマーク用テストデータのセットアップ
  tools: Read, Write, Bash
  # 読み取り制限なし - seed_data.rbにアクセス可能
  ---

  これによりセッション全体のhooksを汚さず、エージェントごとに制約を分離できます
  。先ほどの「Bash経由で迂回」という回りくどい設計が不要になり、かなりクリーンに
  なります。

❯ Bashの権限ってあるとなんでもできちゃう？                                      

⏺ 鋭い指摘です。これは重要な問題です。

  Bashがあると何でもできる問題

  現在の tuning-agent は tools: Read, Edit, Bash, Glob, Grep, Write
  を持っています。

  - Read hook → benchmark/seed_data.rb の Read をブロック
  - でも Bash → cat benchmark/seed_data.rb で普通に読めてしまう

  hooks は ツール単位 でしかかからないので、Bash に対しても hook
  を追加する必要があります。ただし Bash
  の場合、コマンド文字列をパースしてブロックする必要があり、完全な防御は原理的に
  不可能です（変数展開、パイプ、エイリアス等で迂回できる）。

  現実的な選択肢

  ┌─────────────────────────────┬───────────────────────────┬──────────────┐
  │         アプローチ          │          防御力           │    実用性    │
  ├─────────────────────────────┼───────────────────────────┼──────────────┤
  │ A. Bashにもhook追加（パター │ 中（迂回可能だが抑止力に  │ 高           │
  │ ンマッチ）                  │ はなる）                  │              │
  ├─────────────────────────────┼───────────────────────────┼──────────────┤
  │ B. Bashを剥奪 → 必要コマン  │ 高                        │ 中（柔軟性低 │
  │ ドを個別スクリプトに        │                           │ 下）         │
  ├─────────────────────────────┼───────────────────────────┼──────────────┤
  │ C. ソフト制約のみ（現状）   │ 低                        │ 高           │
  ├─────────────────────────────┼───────────────────────────┼──────────────┤
  │ A+C の組み合わせ            │ 中〜高                    │ 高           │
  └─────────────────────────────┴───────────────────────────┴──────────────┘

  B案: Bashを剥奪して専用スクリプトで代替

  一番堅いのはこれです。tuning-agent から Bash
  を外し、必要な操作を専用スクリプト経由にする：

  scripts/
  ├── run-benchmark.sh      ← RAILS_ENV=test ruby benchmark/run.rb
  ├── run-specs.sh           ← bundle exec rspec "$@" --format progress
  ├── run-migration.sh       ← RAILS_ENV=test bin/rails db:migrate
  ├── run-rollback.sh        ← RAILS_ENV=test bin/rails db:rollback
  ├── run-seed.sh            ← RAILS_ENV=test ruby benchmark/setup.rb
  ├── git-commit.sh          ← git add "$@" && git commit -m "$MESSAGE"
  ├── git-reset.sh           ← git reset --hard HEAD~1
  └── git-status.sh          ← git status / git diff / git log

  ただしこれだと
  エージェントの自律性が大きく下がる（想定外の状況に対応できない）。

  A案: Bash hookで抑止

  # Bash PreToolUse hook
  COMMAND="$TOOL_INPUT"  # コマンド文字列
  if echo "$COMMAND" | grep -qE
  '(cat|head|tail|less|more|bat|sed|awk|python|ruby 
  -e).*benchmark/(seed_data|setup|run\.rb)'; then
    echo "BLOCKED: benchmark内部ファイルへのBash経由アクセスは禁止です"
    exit 2
  fi

  迂回できるが、「うっかり読む」を防ぐには十分。

  ---
  どうしますか？個人的には A+C（Bash hook + 
  ソフト制約）で始めて、もし実際にすり抜けが起きたらB（Bash剥奪+専用スクリプト）
  に進化させるのがバランス良いと思いますが、最初からBで固くいくのもありです。

  「不明なエラー」の実態

  autoresearch のループでは、「不明なエラー」に対する方針は明確です：

  ▎ If you can't get things to work after more than a few attempts, give up.

  つまり深い診断は不要。crash として記録し、git reset して次の実験に進む。これが
  ループの強みで、1回の失敗に時間をかけるより次を試す方が生産的です。

  結論

  Bashが本当に必要な場面 = ゼロ（全て事前スクリプト化 or Bash不要）

  設計思想は「デバッグせずに discard

⏺ 現状のホワイトリストは直接実行を前提にしているので、docker compose exec
  付きコマンドは全てブロックされます。

  対応方法として、ホワイトリストのマッチング前に docker compose exec プレフィッ
  クスを剥がす処理を入れるのが一番シンプルです。各guardスクリプトの冒頭で：

  # docker compose exec プレフィックスを除去してからホワイトリストと照合
  COMMAND=$(echo "$COMMAND" | sed -E
  's/^\s*docker\s+compose\s+exec\s+(-e\s+\S+\s+)*\S+\s+//')

  こうすれば docker compose exec -e RAILS_ENV=test app ruby benchmark/run.rb が
  RAILS_ENV=test ruby benchmark/run.rb
  に正規化されてからマッチされます。既存のパターンがそのまま使えます。
