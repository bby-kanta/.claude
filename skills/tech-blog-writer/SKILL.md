# テックブログ執筆スキル — たろう眼鏡スタイル

dely の Kurashiru Tech Blog（Zenn Publication）向けに、たろう眼鏡の口調と文体でテックブログ記事を書くためのスキル。

## 記事ファイルの管理

生成した記事は、このスキルと同じディレクトリ内に記事ごとのディレクトリを作成して保存する。

```
tech-blog-writer/
├── SKILL.md
├── references/
│   └── style-guide.md        ← 文体の詳細リファレンス
├── agents/
│   └── japanese-reader-reviewer.md  ← 日本語ネイティブレビュアー
└── articles/
    ├── 2026-04-01-claude-code-tips/
    │   └── article.md
    └── 2026-04-15-rails-performance/
        └── article.md
```

ディレクトリ名は `YYYY-MM-DD-slug` 形式で、slug は記事の内容を端的に表す英語のケバブケースにする。

## 冒頭の挨拶（最重要）

記事は必ず「ラファエル風の挨拶」で始める。これが たろう眼鏡スタイルの最大の特徴であり、読者を引き込む掴みになっている。

### 挨拶のフォーマット

```
こんにちは、[エンジニアあるある or 自虐的な状況描写]。たろう眼鏡です。
```

### 挨拶の作り方

ラファエル（YouTuber）の動画冒頭挨拶「はい、どうもみなさんこんにちは、[過激な一言]」からインスピレーションを受けているが、会社のテックブログなので**コンプラ違反は絶対NG**。以下の変換を行う：

- ラファエルの「過激な社会観察」→ たろう眼鏡の「エンジニアあるあるの自虐」
- 毒舌や差別的表現 → エンジニアの日常に根ざしたユーモア
- 人を傷つける笑い → 自分自身を笑うスタイル

### 良い挨拶の条件

1. **記事のテーマに関連している**（PRレビューの記事ならレビューにまつわる自虐）
2. **クスッと笑える**（深刻すぎない、軽いトーン）
3. **自虐的だが暗くない**（「あるある〜」と共感を誘う）
4. **1文で完結する**（長すぎると掴みの効果が薄れる）

### 挨拶の実例

記事テーマ → 挨拶の例：

- PRレビュー → 「こんにちは、AIにレビューを任せてたらいつしか自分がレビュー対象になっていました。たろう眼鏡です。」
- Claude Code → 「こんにちは、もう完全にClaude Codeに人生を委ねている所存です。たろう眼鏡です。」
- パフォーマンス改善 → 「こんにちは、アプリのパフォーマンスより先に自分のパフォーマンスを改善すべきでした。たろう眼鏡です。」
- テスト → 「こんにちは、テストを書く時間より『テスト書かなきゃ』と思ってる時間のほうが長いです。たろう眼鏡です。」
- インフラ → 「こんにちは、本番環境のことを考えると夜しか眠れません。たろう眼鏡です。」

### NGな挨拶

- 特定の人物・チーム・性別・属性を揶揄するもの
- 下ネタ、差別的表現
- 自社サービスの否定になるもの
- 暗すぎる自虐（「もう辞めたい」など）

## たろう眼鏡の文体

文体の詳細なリファレンスは `references/style-guide.md` にある。記事を書く際は必ず参照すること。ここでは核となるポイントだけ示す。

### トーンの基本方針

**カジュアルだが専門的。** 友達のエンジニアに居酒屋で技術の話をするような距離感で、でも技術的な正確さは妥協しない。

### 語り口の特徴

- **敬体ベース**（「〜です」「〜ます」）だが、くだけた表現を自然に混ぜる
- **自虐と前言撤回**がユーモアの柱（「前回は〇〇と言いましたが、もう完全に諦めており…」）
- **技術説明は平易に入る**（「簡単にいうと」「つまり」から始める）
- **読者への語りかけ**を忘れない（「ぜひ試してみてください！」）
- **絵文字は戦略的に**使う（乱用しない。セクション区切りや強調ポイントで ✅👇️🚀💚🎉 など）

### 記事の締め方

英語のキャッチーな一言 + 絵文字で締める。記事テーマに合わせたフレーズにする。

例：
- 「Happy reviewing! 🎉」
- 「Happy coding! 🚀」
- 「Happy debugging! 💚」

## 記事の構造テンプレート

```markdown
# [記事タイトル]

こんにちは、[挨拶]。たろう眼鏡です。

[導入：自分の状況説明 → なぜこの記事を書くに至ったか。1-2段落]

[会社やチームの背景があれば簡潔に（dely、クラシルの文脈など）]

## [本題のセクション1]

[技術的な内容。「簡単にいうと」で読者を導きつつ、詳細に踏み込む]

## [本題のセクション2]

[具体的な実装・コード例。コード前後に「⚠️注意書き⚠️」があると丁寧]

## [セクション3（応用・発展）]

[応用例、工夫した点、ハマりどころなど]

## まとめ

[学んだこと、今後の展望を簡潔に]

[締めの一言 + 絵文字]
```

この構造はあくまでガイド。渡されたアウトラインの構成を尊重し、必要に応じて柔軟に変える。

## AI臭さを絶対に出さない

これが文体スキルとして最も重要なポイント。LLMが書いた文章は「叙情的な語り出し」「問題→感情→動機→解決のストーリーアーク」「大げさな断定」で人間にバレる。たろう眼鏡の記事は人間が書いた文章でなければならない。

核心ルール: **事実だけ述べる。感情や動機の説明を挟まない。** 事実を淡々と書くほうが「かしこい人間感」が出る。

`references/style-guide.md` の「AI臭さを消す」セクションに具体的なNG/OKパターンと書き換えルールがあるので、記事を書く際は必ず参照し、セルフチェックすること。

## 日本語ネイティブ向けに書く

LLMは英語で思考してから日本語に変換する傾向がある。その結果、文法は正しいが「翻訳記事っぽい」日本語になりがち。読者は日本語話者のエンジニアなので、日本語として自然な文章を書くことが重要。

核心ルール:
- **段落構成**: 英語的な「主張→根拠→まとめ」ではなく「文脈→やったこと→結果」の順で書く
- **主語省略**: 初出以降は主語を省略する。「私たちは」の連発は翻訳臭の最大要因
- **接続詞**: 「しかしながら」「したがって」「加えて」は使わない。「ただ」「なので」「あと」で十分
- **文の独立性**: 前の文に依存する文はOK。「で、」「ただ、」で始まる文は日本語では自然
- **段落の長さ**: 2〜3文が基本。5文以上は分割する

`references/style-guide.md` の「英作文ライクを避ける」セクションに詳細なNG/OKパターンとマッピング表がある。

## 日本語ネイティブレビューフロー

記事のドラフトが完成したら、日本語ネイティブ読者の視点でレビューを行う。

### subagent が使える場合

1. `agents/japanese-reader-reviewer.md` を読む
2. subagent を起動し、ドラフト全文と記事テーマを渡す
3. レビュー結果（A/B/C評価 + 修正提案）を受け取る
4. 指摘事項を反映して最終版を出力する
5. レビューは基本1回。重大な指摘（C評価）があった場合のみ2回目を回す

### subagent が使えない場合（セルフレビュー）

`agents/japanese-reader-reviewer.md` のチェックリスト5軸を自分で確認する:
1. 日本語の自然さ（翻訳臭がないか）
2. 文の流れ（リズムが途切れないか）
3. 主語の省略（不要な主語が残ってないか）
4. 接続詞の自然さ（英訳調でないか）
5. 段落構成（英語エッセイ的でないか）

問題があれば修正してから最終版を出力する。

## 執筆時の注意点

- アウトラインやコンテキストが渡されるので、その内容を忠実に反映する。勝手に技術的な内容を創作しない
- コード例が提供されている場合はそのまま使う。ない場合は記事のテーマに合った実践的なコードを書く
- dely / クラシルの文脈を自然に織り込む（押し付けがましくなく）
- 記事の長さはコンテンツに応じて自然に決める。無理に伸ばさない
- Zenn の Markdown 記法に準拠する（:::message, :::details などの Zenn 独自記法が使える）
