## LLM Wikiとは何か

### RAGの「毎回やり直し」問題

多くの人がやっているLLM × ドキュメントの使い方は、RAG（Retrieval-Augmented Generation）です。NotebookLM、ChatGPTのファイルアップロード、社内のRAGシステム、だいたいこの構造になっています。

ドキュメントをベクトルDBに入れて、質問が来たら関連チャンクを検索して、LLMに投げて回答を生成する。よく機能します。ただ、蓄積がない。

5つのドキュメントを横断する質問をしたとき、LLMは関連する断片を見つけて、合成して、回答する。同じ質問を明日もう一度したら、まったく同じ検索と合成をゼロからやり直します。前回の回答は消えていて、知識として残っていない。

### 「知識をコンパイルする」という発想

LLM Wikiは構造が違います。

ソースドキュメントを読んで、得た知識を**永続的なWikiとして書き出す**。新しいソースが追加されるたびに、既存のWikiページを更新し、相互参照を張り、矛盾があればフラグを立てる。知識は一度「コンパイル」されたら蓄積されていく。

RAGが「毎回ゼロから検索→合成」なのに対して、LLM Wikiは「一度コンパイル→蓄積→参照」です。相互参照はすでに張られているし、矛盾はすでに検出済みだし、統合もすでに終わっている。

### 3層アーキテクチャ

LLM Wikiは3つのレイヤーで構成されます。

**Raw Sources（ソース）** — ユーザーが集めたドキュメント群。記事、論文、WebクリップなどLLMはこれを読むが、絶対に変更しない。

**Wiki** — LLMが生成・管理するマークダウンファイル群。要約ページ、概念ページ、エンティティページ、比較ページなど。LLMがすべて書いて、すべてメンテナンスする。ユーザーは読むだけ。

**Schema（スキーマ）** — Wikiの構造と規約を定義するドキュメント。ディレクトリ構成、ページフォーマット、操作手順などが書かれている。LLMを野放しにせず、一貫したルールでWikiを管理させるための設定ファイルです。

Karpathy氏の表現を借りれば、「**ObsidianがIDE、LLMがプログラマー、WikiがCodebase**」。コードを書くようにWikiページを書き、Obsidianでリアルタイムに結果を確認する。

### 4つの操作

LLM Wikiには4つの基本操作があります。

**ingest（取り込み）** — メイン操作。新しいソースをLLMに読ませて、Wikiに統合する。ソースを読む→要約ページを作る→既存のWikiページを更新→相互参照を張る→index/logを更新、という流れです。1つのソースが10〜15のWikiページに影響するのは普通で、むしろそうなるべきです。

**query（質問）** — Wikiに対して質問する。LLMがindexを読んで関連ページを探し、内容を読み込んで回答を合成する。RAGと似ていますが、参照先が生のドキュメントではなくすでに構造化されたWikiページなので、回答の質が違います。良い回答はWikiに書き戻して残せます。

**lint（健全性チェック）** — Wikiの品質を定期的に点検する。ページ間の矛盾、孤立ページ（どこからもリンクされていない）、頻繁に言及されるのに独立ページがない概念、などを検出します。

**index + log** — indexはWikiのカタログ、logは操作の時系列記録。LLMはまずindexを読んでWikiの全体像を把握し、そこからページをたどる。logは「最近何をしたか」を把握するための記録です。

### 「各自カスタマイズしろ」

Karpathy氏自身がGistの中でこう書いています。

> This document is intentionally abstract. It describes the idea, not a specific implementation.

ディレクトリ構造もページフォーマットもツールも、ドメインと好みに応じてカスタマイズしろ、というスタンスです。概念だけ伝えるから、あとは自分のLLMエージェントに読ませて一緒に作れ、と。

なので、Claude Codeのスキルとして作ってみました。

## Claude Codeのスキルに落とし込む

KarpathyのGistをそのまま`/skill-creator`に投げました。

`/skill-creator`はClaude Codeのスキル（SKILL.md）を対話的に設計・生成するスキルです。概念文書を読ませると、ユーザーとのQ&Aを通じて具体的なSKILL.mdを生成してくれます。

最初にいくつか聞かれました。

- **ユースケース**: 技術リサーチがメイン。RDB、Railsなどジャンルごとにフォルダを分けたい
- **閲覧ツール**: Obsidian。`[[wikilinks]]`形式とYAMLフロントマターを使う
- **配置場所**: `~/Documents/Obsidian Vault/llm-wiki/`
- **言語**: すべて日本語
- **ソースの場所**: `~/Documents/Obsidian Vault/raw/Clippings/`（Obsidian Web Clipperで取り込んだもの）

これを元に、SKILL.mdが生成されました。引数パースのルール、ディレクトリ構造、ページフォーマット、4つの操作（init/ingest/query/lint）の手順、運用ガイドラインまで一式入っています。

できあがったスキルの基本構造はこうなっています。

```
~/Documents/Obsidian Vault/llm-wiki/
├── index.md                        # 全体カタログ
├── wiki/
│   ├── rdb/                        # ジャンルフォルダ
│   │   ├── index.md                # ジャンル内カタログ
│   │   ├── log.md                  # ジャンル内操作ログ
│   │   ├── _overview.md            # ジャンル概要
│   │   ├── SELECT最適化.md
│   │   └── ...
│   └── rails/
│       └── ...
└── sources/                        # ソースの要約・メタデータ
    ├── rdb/
    └── rails/
```

各WikiページにはYAMLフロントマターがあり、`title`、`genre`、`type`（concept/entity/source-summary/comparison/synthesis）、`sources`、`related`が入ります。ページ間は`[[wikilinks]]`でリンクされ、Obsidianのグラフビューで全体の構造が可視化できます。

実際にingestを回してみると、概念文書を変換しただけでは足りない部分が出てきました。「これ新規ページにするの？既存ページに追記するの？」「indexが肥大化してきたけどどうする？」といった、実運用に必要な判断基準が書かれていない。

## 自分なりのアレンジ — 4つの設計判断

Karpathyの概念文書にはなかった、自分で追加した設計判断を4つ紹介します。

### 1. 2層index/log構造

Karpathyの概念ではindexは1ファイル、logも1ファイルです。Wiki全体のページを1つのindexに全部載せる。

ジャンルが2〜3個、ページが20件くらいのうちは十分ですが、100件を超えてくるとqueryのたびにLLMが巨大なindexを全部読むことになる。コンテキストウィンドウの無駄遣いです。

indexとlogを2層構造にしました。

- **全体index.md** — ジャンル一覧と統計のみ。個別ページは載せない
- **ジャンルindex（`wiki/<genre>/index.md`）** — そのジャンル内の全ページの詳細カタログ
- **ジャンルlog（`wiki/<genre>/log.md`）** — そのジャンル内の操作ログ

LLMはまず全体indexでジャンルを特定し、そこからジャンルindexに飛んで個別ページを探す。2段階のナビゲーションになりますが、読み込むトークン量は大幅に減ります。全体のlog.mdは不要にしました。ジャンル別で管理すれば十分です。

### 2. ソース原本のジャンル別自動整理

Obsidian Web ClipperでWebページを保存すると、`Clippings/`直下にフラットに溜まっていきます。最初は数件なので問題ないですが、20件、30件と増えるとソースを探すのが面倒になる。

ingestの手順に「ソース原本をジャンル別フォルダに移動する」ステップを追加しました。

MySQLの記事をingestしたら、`Clippings/MySQL...md`が`Clippings/rdb/MySQL...md`に自動で移動されます。ソース要約ページの原本パスも更新される。Wikiだけでなくソース側も整理された状態が保たれます。

LLMがingestのついでにやってくれるので、人間の手間はゼロです。Karpathyの概念文書には「ソースは不変」とだけ書かれていますが、「ソースの場所を整理する」ことは内容の変更ではないので問題ありません。

### 3. ページの作成・更新・分割の判断基準

ingestで最も重要な判断は「新しいページを作るか、既存ページを更新するか」です。Karpathyの概念文書にはこの基準がありません。

LLMに暗黙の判断を任せると、セッションによって挙動がブレます。あるときは1ソースから10ページ作り、別のときは3ページしか作らない。SKILL.mdに明文化した判断基準を組み込みました。

**新規ページを作るとき:**
- そのコンセプトがまだWikiに独立ページとして存在しない
- 既存ページの1セクションに収まらないほどの情報量がある
- 複数のページから参照されそうな概念（例: 「定数畳み込み」がWHERE句最適化とRange最適化の両方から参照される）

**既存ページを更新するとき:**
- 新しいソースが既存ページのトピックに直接関連する追加情報を含む
- 既存の記述を補強・修正・矛盾指摘する情報がある

**ページを分割するとき:**
- 既存ページの1セクションが肥大化して、独立したトピックとして成立する場合
- 分割元には要約を残して`[[新ページ]]`でリンクする

**横断ページ（これがWikiの真価）:**
- 複数のソースにまたがる知識を1つのページにまとめる。例えば「インデックス戦略」ページがSELECT最適化、WHERE句最適化、Range最適化の3ソースすべてから得たインデックス関連の知識を統合する

スキーマに書いておけば、セッションをまたいでも一貫した粒度でページが作られます。

### 4. query結果をWikiに保存する/しないの判断基準

Karpathy氏は「良い回答はWikiに保存できる」と書いています。とはいえ、何が「良い回答」なのかの基準は示されていません。

基準がないと、「全部保存」か「全部保存しない」のどちらかに偏ります。全部保存するとWikiが肥大化して質が下がるし、全部保存しないとqueryで得た知識が消えてもったいない。

SKILL.mdに以下の判断基準を追加しました。

**保存すべきケース:**
- Wikiにまだない新しい切り口や視点を含む回答（例: 複数概念を独自に比較した分析）
- 新しいコンセプトを導入している
- 繰り返し参照される可能性が高い汎用的な分析

**保存不要なケース:**
- 既存ページの知識を組み合わせただけで、新しい知識を追加していない
- 特定のクエリやコードなど、個別具体的すぎて再利用性が低い
- 既存ページに追記すれば済む程度の内容

queryのたびにLLMが「この回答はWikiに保存しますか？」と聞いてくるようにしてあり、上の基準に基づいて提案理由も説明します。最終判断はユーザーが行います。

## まとめ

LLM Wikiは知識を一度コンパイルして蓄積するアプローチで、RAGとは別物です。

Karpathyの概念文書が抽象的なのは意図的で、そのぶんカスタマイズの余地が大きい。今回紹介した4つの設計判断（2層index、ソース整理、ページ判断基準、query保存基準）は自分のユースケースに合わせたものなので、別のドメインや別のワークフローなら違う判断になると思います。

興味がある方は、[KarpathyのGist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)をそのままLLMに投げるだけで出発点は作れます。自分はRDBまわりのリサーチで使い始めたところなので、運用が進んだらまた書きます。

Happy wiki-ing! 🚀
